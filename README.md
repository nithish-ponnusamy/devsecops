# DevSecOps CI/CD Pipeline

## Enterprise DevSecOps CI/CD Pipeline with Jenkins, Kubernetes, SonarQube, Snyk, Docker & Trivy

---

## What This Project Is

A **Task Manager REST API** built with Spring Boot and MongoDB, wrapped in a complete **DevSecOps CI/CD pipeline** that builds, scans for security vulnerabilities, containerizes, and deploys the application automatically.

---

## Architecture Flow

```
Developer pushes code to GitHub
         |
         v
  Jenkins detects the commit
         |
         v
  Slack Notification: "Pipeline Started"
         |
         v
  +-------+------+----------+------+--------+-------+--------+
  | Build  | Test | SonarQube | Snyk | Docker | Trivy | Deploy |
  +-------+------+----------+------+--------+-------+--------+
         |
         v
  Slack Notification: "Pipeline Succeeded/Failed"
```

## Part 1: The Application

The app is a REST API with the following source files:

```
src/main/java/com/devsecops/app/
├── DevSecOpsApplication.java        --> Entry point, starts Spring Boot
├── model/Task.java                  --> Data model (id, title, status, priority, etc.)
├── repository/TaskRepository.java   --> MongoDB queries (auto-generated by Spring)
├── service/TaskService.java         --> Business logic (CRUD operations)
└── controller/
    ├── TaskController.java          --> API endpoints under /api/v1/tasks
    └── HomeController.java          --> Root endpoint (/) showing available routes
```

### How Data Flows Through the App

```
HTTP Request
    |
    v
TaskController (receives request, validates route)
    |
    v
TaskService (applies business logic, sets timestamps)
    |
    v
TaskRepository (translates to MongoDB queries)
    |
    v
MongoDB (stores/retrieves JSON documents in "tasks" collection)
    |
    v
Response returned as JSON
```

### API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/` | Application info and endpoint directory |
| GET | `/api/v1/tasks/health` | Health check |
| GET | `/api/v1/tasks` | List all tasks |
| GET | `/api/v1/tasks/{id}` | Get task by ID |
| POST | `/api/v1/tasks` | Create a new task |
| PUT | `/api/v1/tasks/{id}` | Update a task |
| DELETE | `/api/v1/tasks/{id}` | Delete a task |
| GET | `/api/v1/tasks/status/{status}` | Filter tasks by status |
| GET | `/api/v1/tasks/priority/{priority}` | Filter tasks by priority |
| GET | `/api/v1/tasks/assignee/{name}` | Filter tasks by assignee |
| GET | `/actuator/health` | Spring Actuator health with MongoDB status |

### Configuration

The application is configured via `src/main/resources/application.yml`:

- Reads `MONGODB_URI` from environment variable (defaults to `mongodb://localhost:27017/devsecops`)
- Runs on port 8080
- Exposes Spring Actuator health, info, and metrics endpoints

## Part 2: Containerization

### Dockerfile (Two-Stage Build)

```
Stage 1 (Build):
  maven:3.9.6-alpine image
      |
      v
  Copies pom.xml --> downloads all dependencies (cached layer)
      |
      v
  Copies source code --> runs "mvn clean package"
      |
      v
  Produces: target/devsecops-app.jar

Stage 2 (Runtime):
  eclipse-temurin:17-jre-alpine (lightweight, ~100MB)
      |
      v
  Creates non-root user "appuser" (security best practice)
      |
      v
  Copies only the .jar from Stage 1 (no source code, no Maven)
      |
      v
  Runs as appuser on port 8080 with health check
```

The two-stage build means the final image is small and contains no build tools or source code.

### Docker Compose

Docker Compose runs two containers together:

```
+----------------------------------------------+
|  Docker Compose Network                       |
|                                               |
|  +---------------+    +-------------------+   |
|  |  mongodb       |    |  app              |   |
|  |  mongo:7.0     |<---|  devsecops-app    |   |
|  |  port: 27017   |    |  port: 8085->8080 |   |
|  |  volume: data  |    |  env: MONGODB_URI |   |
|  +---------------+    +-------------------+   |
+----------------------------------------------+
```

The app container connects to MongoDB using Docker's internal DNS name `mongodb` (the service name in docker-compose.yml).

## Part 3: CI/CD Pipeline (Jenkinsfile)

The pipeline has 12 stages that run sequentially:

```
Stage 1:  Notify Start
          --> Calls slackNotification('STARTED')
          --> Team sees: "Pipeline Started - Job #42"

Stage 2:  Checkout
          --> Pulls latest code from GitHub

Stage 3:  Build
          --> Runs: mvn clean package -DskipTests
          --> Compiles Java source into .jar

Stage 4:  Unit Tests
          --> Runs: mvn test
          --> Publishes JUnit test results to Jenkins

Stage 5:  SonarQube Analysis
          --> Sends code to SonarQube server
          --> Scans for: bugs, code smells, vulnerabilities, duplication

Stage 6:  Quality Gate
          --> Waits for SonarQube to return pass/fail
          --> If FAIL: pipeline aborts

Stage 7:  Snyk Security Scan
          --> Scans pom.xml dependencies for known CVEs
          --> Example: "log4j 2.14.1 has CRITICAL vulnerability"
          --> Continues on medium-level findings

Stage 8:  Docker Build
          --> Builds container image using the Dockerfile
          --> Tags as: yourdockerhubuser/devsecops-app:BUILD_NUMBER

Stage 9:  Trivy Image Scan
          --> Scans the Docker image for OS/library vulnerabilities
          --> Reports HIGH and CRITICAL findings

Stage 10: Push to DockerHub
          --> Logs into DockerHub with stored credentials
          --> Pushes tagged image + latest tag

Stage 11: Deploy to Kubernetes
          --> Applies K8s manifests (namespace, configmap, secret, mongodb, app)
          --> Updates deployment with new image tag
          --> Waits for rollout to complete

Stage 12: Trivy K8s Scan
          --> Scans K8s YAML manifests for misconfigurations
          --> Example: "container running as root", "no resource limits"
```

**Post actions** (run regardless of result):
- Sends Slack notification with SUCCESS or FAILURE
- Cleans Jenkins workspace and removes local Docker images

## Part 4: Security Scanning (DevSecOps)

Security is checked at **4 different points** in the pipeline:

```
Code Level:
  SonarQube ----> Static analysis of Java source code
                  Finds: SQL injection patterns, hardcoded secrets,
                         null pointer risks, code duplication

Dependency Level:
  Snyk ----> Scans third-party libraries in pom.xml
             Finds: Known CVEs in Spring, MongoDB driver, Jackson, etc.

Container Level:
  Trivy (image scan) ----> Scans the built Docker image
                           Finds: Vulnerable OS packages in Alpine,
                                  outdated JRE components

Infrastructure Level:
  Trivy (config scan) ----> Scans Kubernetes YAML files
                            Finds: Missing resource limits,
                                   containers running as root,
                                   missing security contexts
```

```
    +-- NodePort Service (external access on port 30080)
    +-- Readiness probe: /api/v1/tasks/health
    +-- Liveness probe: /api/v1/tasks/health
    +-- Security: runAsNonRoot, readOnlyRootFilesystem
```

## Part 6: Jenkins Shared Library

Instead of placing Slack logic directly in every Jenkinsfile, the notification code is in a reusable shared library:

```
jenkins-shared-library/vars/slackNotification.groovy
```

This function is called as `slackNotification('SUCCESS')` and:

1. Maps status to a color and emoji (green for success, red for failure)
2. Builds a message with job name, build number, branch, and duration
3. Sends to the `#devsecops-pipeline` Slack channel

Any Jenkins pipeline in the organization can reuse this by adding `@Library('jenkins-shared-library') _` at the top of the Jenkinsfile.

## Quick Start (Run Locally)

```bash
# Start the application with Docker Compose
docker compose up --build -d

# Check container status
docker compose ps

# Test the API
curl http://localhost:8085/
curl http://localhost:8085/api/v1/tasks/health
curl http://localhost:8085/api/v1/tasks

# Create a task
curl -X POST http://localhost:8085/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"My Task","description":"Task description","priority":"HIGH","assignedTo":"team"}'

# Stop the application
docker compose down
```

## Cleanup

```bash
# Stop and remove containers
docker compose down

# Remove volumes (deletes MongoDB data)
docker compose down -v

# Remove Docker images
docker rmi devsecops-app:latest
```
